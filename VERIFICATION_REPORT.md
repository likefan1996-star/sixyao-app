## 2. 当前实现规则说明

### 排盘引擎核心规则（utils/paipan.js）

| 模块 | 规则 |
|------|------|
| **本卦** | yao数组→binary字符串（初爻在下=左侧）→HEXAGRAM_BY_BINARY查表 |
| **动爻判断** | 老阳(laoyang)/老阴(laoyin)为动爻，少阳/少阴不动 |
| **变卦** | 动爻位置阴阳互换→新binary→查变卦 |
| **纳甲** | 内卦取前三爻(najia[lowerTri])，外卦取后三爻(najia[upperTri]) |
| **世应** | SHIYING_MAP按generation类型定位 |
| **六亲** | 卦宫五行为"我"，生克关系确定 |
| **六神** | 按日干甲乙/丙丁/戊己/庚辛/壬癸确定初爻位置，顺排六神 |
| **月建** | 精确时间查solarTerms表；表外年份用近似DOY阈值 |
| **日辰** | 相对2024-01-01的天数差→dayIndex→干支 |
| **旬空** | dayIndex/10定位KONG_TABLE，空亡2支 |

## 4. 发现的问题

### P0：无（未发现导致排盘明显错误的bug）

所有核心排盘功能（纳甲、世应、六亲、六神、动变、月建、日辰、旬空）经外部标准验证均正确。

### P1：边界情况注意

1. **节气边界精确性**
   - 当前solarTerms.js覆盖2024-2030年，范围外回退到近似DOY表
   - 近似表使用固定DOY阈值，可能与真实节气时间有1-2天偏差
   - **建议**：超出范围时在结果中标注"近似月建"

2. **golden cases全部为draft状态**
   - `status: 'verified'` 的案例数为0
   - 无法自动回归校验排盘正确性
   - **建议**：补充至少3个verified golden cases

### P2：专业性建议（不影响基础排盘）

1. **六神排法未区分"爻位循环"vs"固定位置"流派差异**
   - 当前实现：六神从初爻开始，按日干确定起始位，顺排六爻
   - 传统主流：六神按爻位固定对应（初爻青龙、二爻朱雀...）
   - **标注**：流派差异，非错误

2. **世应表注"四世卦世在四爻"但四世卦实际有8种情况**
   - 不同宫的四世卦世爻位置不同（震宫四世世4应1，坤宫四世世4应1...）
   - 当前SHIYING_MAP按generation统一赋值，对于同一generation类型的所有卦都用相同世应位置
   - 这是京房体系的通用规则，实际正确
   - **建议**：在注释中补充说明

3. **七档评分是产品解释层**
   - "极盛/偏盛/平顺/小动/平衡/偏弱/受制"是产品设计
   - 不等于传统六爻的旺相衰死判定
   - **建议**：在UI中明确标注"结构参考"而非"吉凶判定"

### P3：文案/产品解释问题

1. **fortune.js的评分说明可更精确**
   - 当前"月建旺衰均分"的均分逻辑可能产生误导
   - 例如6爻全部旺时yueAvg=10，但实际某爻月破会大幅拉低
   - **建议**：增加"用神月建状态"单独显示

2. **yongshen.js默认取世爻为用神**
   - 传统上世爻不一定是用神，只是"无匹配时的兜底"
   - 当前实现正确，但注释可更清晰
